# Business Logic

## Основные принципы

1. Google Calendar является источником истины для расписания.
2. Каждая встреча хранится как отдельный `instance` в `calendar_event_instances`.
3. Статус оплаты хранится на уровне конкретной встречи (`instance`), а не клиента в целом.
4. Очередь `notification_jobs` задаёт срок и статусы payment reminder на backend; **текст и отправка** напоминания в Telegram — в **конструкторе ботов** (см. `docs/automation.md` §7.6 и `docs/constructor_scenarios_api.md`).
5. Backend является оркестратором бизнес-логики, интеграций и фоновых процессов.

Источники истины:

- расписание: Google Calendar
- данные системы: PostgreSQL

---

## Главные сущности в бизнес-процессах

- `therapists` — терапевты и их глобальные настройки reminder
- `clients` — клиенты терапевтов
- `client_chat_links` — связь Telegram group chat с клиентом
- `calendar_events` — родительские события календаря
- `calendar_event_instances` — операционные записи реальных встреч
- `notification_jobs` — очередь отправки уведомлений

---

## Пользовательские сценарии

## 1. Подключение системы и стартовая настройка терапевта

Терапевт:

- запускает Telegram-бота и проходит стартовую настройку:
  - подключает Google Calendar (OAuth через backend, см. `docs/integrations.md`)
  - задаёт локальное время для расчёта напоминаний одним из способов (конструктор передаёт в backend `POST .../therapists/upsert-profile`):
    - **город** (свободный текст из ответа в боте): backend по геокодингу и координатам определяет IANA-таймзону и сохраняет её в `therapists.timezone`;
    - либо **явная таймзона IANA** (если сценарий бота или оператор передают поле `timezone` напрямую; из данных Telegram таймзона пользователя не определяется надёжно)
  - **напоминания об оплате**: задаёт `payment_reminder_timing` (`before` / `after`) и `payment_reminder_offset_minutes` (смещение в минутах); позже может перенастроить в меню бота
  - добавляет бота в общий чат с клиентом, чтобы связка терапевт-клиент фиксировалась сразу в нужном контексте

Результат:

- система готова к синхронизации встреч и отправке уведомлений.
- backend сохранил профиль терапевта в PostgreSQL.
- backend получил подключение к Google Calendar и может запускать синхронизацию.

## 1.0 Меню бота (терапевт)

В меню доступны как минимум:

- перенастройка схемы payment reminder (`payment_reminder_timing`, `payment_reminder_offset_minutes`)
- **клиенты и напоминания**: список всех клиентов терапевта с меткой, включены ли для клиента напоминания об оплате (`send_payment_reminders`); терапевт может включить или выключить напоминания для каждого клиента

---

## 1.1 Привязка чата клиента к клиенту и терапевту

1. Терапевт добавляет бота в общий чат с клиентом.
2. Конструктор получает событие добавления бота в чат и отправляет его в backend.
3. Backend создает или находит запись в `clients` для этого терапевта.
4. Backend создает или обновляет запись в `client_chat_links`:
  - `group_chat_id`
  - `selected_therapist_id` = терапевт, в чей чат был добавлен бот
  - `client_id` = id найденного или созданного клиента
5. После этого чат считается привязанным к клиенту
6. Флаг `send_payment_reminders` для клиента далее управляется терапевтом через меню бота.

## 1.2 Привязка клиента к встрече (calendar_events)

Терапевт (через меню **клиенты и напоминания**):

1. После выбора клиента терапевт запрашивает список встреч, доступных для привязки.
2. Конструктор вызывает backend, а backend возвращает список **встреч (`calendar_events`)** без повторов recurring, где:
  - `calendar_events.client_id IS NULL` (встреча еще не привязана)
  - существует как минимум один связанный `calendar_event_instances`, у которого `actual_start_at > now()` (только будущие встречи)
  - в списке показывается название встречи: поле `calendar_events.summary` (в API списка отдаётся как `title`)
3. Терапевт выбирает конкретную встречу.
4. Backend атомарно обновляет связку клиент-встреча в БД:
  - выбрать можно только новую встречу, у которой `calendar_events.client_id IS NULL`
  - если у клиента уже была привязана другая встреча через этот сценарий, backend снимает `client_id` с прежней встречи
  - затем backend записывает `client_id` в новую встречу
5. Таким образом перепривязка выполняется через тот же пользовательский сценарий.
6. После привязки или перепривязки backend пересчитывает связанные `notification_jobs`.

Следствие для reminder’ов:

- Напоминания строятся через `notification_jobs` на уровне `calendar_event_instances`, но релевантность reminder’ов зависит от `calendar_events.client_id`.
- Поэтому при изменении `calendar_events.client_id` backend и/или PostgreSQL-триггер пересчитывает связанные `notification_jobs` для инстансов соответствующей встречи.

## 2. Синхронизация встреч

1. Backend запускает сценарий синхронизации только для **активных** терапевтов.
2. Backend получает события Google Calendar.
3. Для событий и серий backend создает или обновляет записи:
  - `calendar_events`
  - `calendar_event_instances`
4. После обновления встреч backend пересчитывает `notification_jobs`.

Правило:

- система не редактирует события в Google Calendar.

## Активность терапевта (`therapists.is_active`)

- `is_active = false`, если терапевт удалил личный чат с ботом или прекратил подписку на сервис (событие фиксируется сценарием конструктора и передаётся в backend через `PATCH /api/v1/bot/therapists/{therapist_id}`, см. `docs/automation.md` §9.5b).
- По неактивному терапевту **не выполняются**: синхронизация календаря, создание payment reminder jobs, прочие исходящие автоматизации для него и его клиентов.

## Активность клиента (`clients.is_active`)

- Рекомендуется оставить флаг: при прекращении работы с клиентом терапевт деактивирует клиента — напоминания для него отключаются явно, даже если в Google Calendar остались будущие события с привязкой к этому клиенту. Смена флага — через `PATCH /api/v1/bot/clients/{client_id}` (см. `docs/automation.md` §9.5a).
- Если клиент деактивирован только «логически», а встреч больше нет, напоминаний всё равно не будет; флаг даёт предсказуемое поведение при «хвостах» в календаре.

## 3. Напоминание клиенту об оплате

1. Для встречи определяется необходимость reminder:
  - терапевт **активен**, клиент **активен**
  - `send_payment_reminders = true` у клиента
  - глобальные настройки терапевта (`payment_reminder_timing`, `payment_reminder_offset_minutes`)
2. Запись в `notification_jobs` создаётся или обновляется backend-логикой при изменении встречи, клиента, настроек reminder или статуса оплаты.
3. Если встреча перенесена, `scheduled_for` пересчитывается для того же `calendar_event_instance_id`.
4. Если встреча отменена или оплачена до отправки, reminder переводится в `cancelled`.
5. Инстанс считается отмененным, если `calendar_event_instances.status` (case-insensitive) равен `cancelled/canceled/cancel`.
6. Момент `scheduled_for` считается **от границ встречи в календаре** (`actual_start_at` / `actual_end_at` в UTC) и настроек before/after + offset; **таймзона терапевта в профиле и таймзона клиента на этот момент не влияют**. Сообщение об оплате адресовано **клиенту** (общий чат); текст и отправка в Telegram — в **конструкторе ботов**, backend отдаёт due-задачи и принимает `mark-sent` / `mark-failed` (см. `docs/automation.md` §7.6 и §9.10–9.12).
7. Клик клиента по кнопке оплаты обрабатывается через backend API (см. сценарий 4).

## 4. Подтверждение оплаты клиентом

1. Клиент нажимает кнопку подтверждения оплаты в Telegram.
2. Конструктор отправляет действие в backend API.
3. Backend идемпотентно обновляет `calendar_event_instances`:
  - `is_paid = true`
  - `payment_confirmed_at = now()`
4. Если reminder по этой встрече ещё не отправлен, backend отменяет или пересчитывает соответствующую запись в `notification_jobs`.

## Правила напоминаний об оплате

- Настройка reminder задается на уровне терапевта и едина для всех его клиентов (per-client только вкл/выкл).
- Допустимые режимы:
  - `before` + `offset` (например, `before + 60`)
  - `after` + `offset` (например, `after + 0`)
- На уровне клиента:
  - `send_payment_reminders = true/false` — только терапевт меняет через бот (список клиентов).

## Границы логики первой версии

Не реализуются:

- бронирование встреч клиентом
- перенос встреч клиентом
- собственная платежная система
- CRM-функции
- сложная аналитика
- multi-therapist управление

---

## Что обязательно реализовать в backend

Минимальный набор доменной логики backend:

1. CRUD для терапевта и его настроек, включая смену `therapists.is_active` и `clients.is_active` через HTTP API (см. `docs/automation.md`).
2. Получение URL Google OAuth и обработка callback; хранение OAuth-учётных данных в **зашифрованном** виде в PostgreSQL.
3. Full sync и incremental sync Google Calendar (в т.ч. заполнение `calendar_events.summary` из Google).
4. Автоматическое создание `client_chat_links` при добавлении бота в чат.
5. Создание `clients` и переключение `send_payment_reminders`.
6. Выдача списка доступных встреч для привязки клиента.
7. Атомарная привязка `client_id` к `calendar_events`.
8. Подтверждение оплаты и отмена reminder.
9. Планирование payment reminder в `notification_jobs` (расчёт `scheduled_for`, статусы), выдача due-задач конструктору и приём `mark-sent` / `mark-failed`; **текст сообщения и отправка в Telegram** — ответственность конструктора ботов.
