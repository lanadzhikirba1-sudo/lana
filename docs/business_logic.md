# Business Logic

## Основные принципы

1. Google Calendar является источником истины для расписания.
2. Каждая встреча хранится как отдельный `instance` в `calendar_event_instances`.
3. Статус оплаты хранится на уровне конкретной встречи (`instance`), а не клиента в целом.
4. Уведомления отправляются через очередь задач `notification_jobs`.
5. Make является оркестратором, но не хранит состояние и не является источником истины.

Источники истины:
- расписание: Google Calendar
- данные системы: Supabase

---

## Главные сущности в бизнес-процессах

- `therapists` — терапевты и их глобальные настройки reminder/digest
- `clients` — клиенты терапевтов
- `client_chat_links` — связь Telegram group chat с клиентом
- `calendar_events` — родительские события календаря
- `calendar_event_instances` — операционные записи реальных встреч
- `notification_jobs` — очередь отправки уведомлений

---

## Пользовательские сценарии

## 1. Подключение системы и стартовая настройка терапевта

Терапевт:
- запускает Telegram-бота и проходит стартовую настройку:
  - подключает Google Calendar (OAuth, см. `docs/integrations.md`)
  - указывает таймзону (IANA), если ещё не зафиксирована
  - **digest**: время ежедневного digest — по умолчанию 09:00 локально в его таймзоне; при необходимости задаёт другое время (то же можно изменить позже в меню)
  - **напоминания об оплате**: задаёт `payment_reminder_timing` (`before` / `after`) и `payment_reminder_offset_minutes` (смещение в минутах); позже может перенастроить в меню бота

Результат:
- система готова к синхронизации встреч и отправке уведомлений.

---

## 1.0 Меню бота (терапевт)

В меню доступны как минимум:
- смена времени daily digest (`digest_time_local`)
- перенастройка схемы payment reminder (`payment_reminder_timing`, `payment_reminder_offset_minutes`)
- **клиенты и напоминания**: список всех клиентов терапевта с меткой, включены ли для клиента напоминания об оплате (`send_payment_reminders`); терапевт может включить или выключить напоминания для каждого клиента

---

## 1.1 Привязка группового чата к клиенту и терапевту

1. Клиент пишет в групповой чат и выбирает терапевта.
2. Создается или обновляется запись в `client_chat_links`:
   - `group_chat_id`
   - `selected_therapist_id` = терапевт, выбранный клиентом
   - `client_id` = `null` (до подтверждения)
3. Терапевту отправляется запрос на подтверждение клиента.
4. Если терапевт подтверждает:
   - создается (или находится) запись в `clients` с `therapist_id = selected_therapist_id`
   - в `client_chat_links.client_id` записывается id этого клиента
   - флаг `send_payment_reminders` выставляется по решению терапевта (в т.ч. через последующий флоу «клиенты и напоминания»)
5. Если терапевт отклоняет:
   - `client_id` остается `null`
   - клиенту предлагается выбрать другого терапевта

## 1.2 Привязка клиента к встрече (calendar_events)

Терапевт (через меню **клиенты и напоминания**):
1. После выбора клиента терапевт запрашивает список встреч, доступных для привязки.
2. Make подтягивает список **встреч (calendar_events)** без повторов recurring, где:
   - `calendar_events.client_id IS NULL` (встреча еще не привязана)
   - существует как минимум один связанный `calendar_event_instances`, у которого `actual_start_at > now()` (только будущие встречи)
3. Терапевт выбирает конкретную встречу.
4. Make записывает связку в БД:
   - `UPDATE calendar_events SET client_id = :client_id WHERE id = :calendar_event_id AND client_id IS NULL`
5. Если обновлено 0 строк (встреча успела стать занятой/привязанной не через этот flow), терапевту показывается сообщение о том, что встреча уже привязана, и предлагается повторно открыть список.

Ограничение перепривязки:
1. Для терапевта действие “перепривязать” недоступно (опция показывается только для `calendar_events.client_id IS NULL`).
2. Перепривязка возможна только в админ-режиме бота (через отдельную команду/сообщения администратора), где backend допускает изменение `calendar_events.client_id`.

Следствие для reminder’ов:
- Напоминания строятся через `notification_jobs` на уровне `calendar_event_instances`, но релевантность reminder’ов зависит от `calendar_events.client_id`.
- Поэтому при изменении `calendar_events.client_id` Supabase пересчитывает (через триггер) связанные `notification_jobs` для инстансов соответствующей встречи.

## 2. Синхронизация встреч

1. Make запускает сценарий синхронизации (только для **активных** терапевтов, см. ниже).
2. Система получает события Google Calendar.
3. Для событий/серий создаются или обновляются записи:
   - `calendar_events`
   - `calendar_event_instances`

Правило:
- система не редактирует события в Google Calendar.

## Активность терапевта (`therapists.is_active`)

- **`is_active = false`**, если терапевт удалил личный чат с ботом или прекратил подписку на сервис.
- По неактивному терапевту **не выполняются**: синхронизация календаря, создание payment reminder jobs, отправка digest, прочие исходящие автоматизации для него и его клиентов.

## Активность клиента (`clients.is_active`)

- Рекомендуется оставить флаг: при прекращении работы с клиентом терапевт деактивирует клиента — напоминания и учёт в digest для него отключаются явно, даже если в Google Calendar остались будущие события с привязкой к этому клиенту.
- Если клиент деактивирован только «логически», а встреч больше нет, напоминаний всё равно не будет; флаг даёт предсказуемое поведение при «хвостах» в календаре.

## 3. Напоминание клиенту об оплате

1. Для встречи определяется необходимость reminder:
   - терапевт **активен**, клиент **активен**
   - `send_payment_reminders = true` у клиента
   - глобальные настройки терапевта (`payment_reminder_timing`, `payment_reminder_offset_minutes`)
2. Запись в `notification_jobs` создаётся/обновляется автоматически в Supabase (v1) триггером на `calendar_event_instances`, поэтому при переносе/изменении встречи планировщик reminder синхронизируется по тому же `calendar_event_instance_id` (пересчитывается `scheduled_for`, а при отмене/оплате reminder переводится в `cancelled`, если он ещё не был отправлен).
   Триггер считает инстанс отменённым, если `calendar_event_instances.status` (case-insensitive) равен `cancelled/canceled/cancel`.
3. В нужное время Supabase-воркер/Edge Function по расписанию забирает `notification_jobs` в статусе `pending` и с наступившим `scheduled_for`, отправляет сообщение в Telegram и обновляет `status`/`sent_at`.
Клик клиента по кнопке оплаты обрабатывается отдельно в Make через webhook от Telegram (см. сценарий 4).

## 4. Подтверждение оплаты клиентом

1. Клиент нажимает кнопку подтверждения оплаты в Telegram.
2. Make принимает webhook от Telegram.
3. Обновляется `calendar_event_instances`:
   - `is_paid = true`
   - `payment_confirmed_at = now()`

## 5. Daily digest для терапевта

1. В заданное локальное время (`digest_time_local` в таймзоне терапевта, по умолчанию 09:00) для **активных** терапевтов формируется digest.
2. В digest включаются встречи и статусы оплат.
3. Сообщение отправляется терапевту в Telegram.

---

## Правила напоминаний об оплате

- Настройка reminder задается на уровне терапевта и едина для всех его клиентов (per-client только вкл/выкл).
- Допустимые режимы:
  - `before` + `offset` (например, `before + 60`)
  - `after` + `offset` (например, `after + 0`)
- На уровне клиента:
  - `send_payment_reminders = true/false` — только терапевт меняет через бот (список клиентов).

## Границы логики первой версии

Не реализуются:
- бронирование встреч клиентом
- перенос встреч клиентом
- собственная платежная система
- CRM-функции
- сложная аналитика
- multi-therapist управление
